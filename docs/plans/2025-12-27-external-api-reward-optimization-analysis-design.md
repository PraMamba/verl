## 1. 任务目标

系统性分析 verl 框架在使用外部 API（如 GPT-4o、Claude 3.5 Sonnet）作为 LLM-as-a-Judge 进行 GRPO 训练时，是否存在针对"时延墙"和"成本挑战"的工程深度优化。

### 核心问题

**时延墙 (Latency Wall)**:
- 外部 API 响应时间：1-5 秒
- GPU 生成速度：毫秒级
- 矛盾：GPU 算力被 I/O 阻塞，训练效率低下

**成本挑战 (Cost Challenge)**:
- API 调用费用可能超过 GPU 租赁费用
- 需要优化调用次数和策略

### 3.1 数据流视角

跟随一个 GRPO 训练步的完整生命周期：

```
Rollout (生成响应)
    ↓
Reward Calculation (外部 API 调用)
    ↓
```

在每个阶段识别：

- 同步/异步机制
- 批处理策略

## 9. 潜在发现预期

基于 verl 的架构特点（Hybrid-Controller + Ray），可能的发现包括：

**可能存在的优化**:
- ✅ 并发控制：Ray actors 天然支持并发执行
- ✅ 批处理：Rollout 阶段的批处理可能影响 reward 批次
- ⚠️ 异步调用：需要检查 reward workers 是否使用 async/await

**可能缺失的优化**:
- ❌ 缓存机制：通用 RL 框架可能不内置 API 缓存
- ❌ 降级策略：可能没有自动降级逻辑
- ❌ 请求合并：可能没有针对外部 API 的请求去重

**需要重点关注**:
- Ray 的异步特性是否被充分利用
- Reward workers 的调度策略
- 配置文件中是否有隐藏的优化开关

（注：以上仅为假设，实际分析需以源码为准）
